# Web ↔ Telegram 主线镜像插件计划书

负责人：苏雨  
执行工程师：砚秋（Codex 工程师）  
状态：计划阶段 / 可进入实现

## 1. 项目背景

当前 OpenClaw 已完成：

- Telegram 代理出站可用
- `session.dmScope` 已切换为 `main`
- Web 端与 Telegram 端已汇入同一条主会话主线

但仍存在一个体验问题：

- Telegram 发来的消息会进入主会话，因此 Web 端可以看到它们
- Web 端发出的消息不会自动镜像发送到 Telegram
- 结果是“同一大脑、不同显示层”，而不是“多端同步聊天”

雨宿希望：

- Web 端仍然保留为重要控制层
- Telegram 端保留为日常聊天入口
- 二者在主会话上尽可能像“同一条连续聊天”

## 2. 项目目标

实现一个 **最小可用的一向镜像插件**：

> 当主会话中的回复来自 Web / Control UI 时，自动镜像一份到 Telegram 私聊端。

这不是全平台群发系统，也不是双向全量桥接；它的第一阶段目标只补齐：

- **Web → Telegram 镜像**

## 3. 范围定义

### 3.1 本期包含（In Scope）

1. 仅针对 **主会话 `agent:main:main`** 生效
2. 仅针对 **私聊主线** 生效，不处理群聊
3. 仅做 **assistant 输出镜像**，不处理用户输入镜像
4. 仅做 **Web / Control UI → Telegram** 的一向镜像
5. 插件需避免重复发送 / 回声循环
6. 插件要支持未来通过配置开关启停

### 3.2 本期冻结（Out of Scope）

1. 不做 Telegram → Web 的强制回推（Web 已可见 transcript）
2. 不做 Discord / Slack / WhatsApp 等多平台统一镜像
3. 不做媒体文件、按钮、富卡片的镜像
4. 不做历史消息回填
5. 不改 OpenClaw 核心源码逻辑，优先使用插件机制
6. 不做群组/Topic 级复杂路由

## 4. 设计原则

1. **最小侵入**：优先使用 OpenClaw 插件 SDK / hook，不魔改 core
2. **单向闭环**：先补 Web → Telegram，不追求一步到位
3. **防回声**：镜像消息不能再次触发镜像链路
4. **主线优先**：一切以 `main` 主会话为准
5. **配置显式**：镜像目标、启用开关、过滤条件必须可配置
6. **低资源友好**：考虑当前服务器资源紧张，逻辑要轻

## 5. 可能实现路径

根据 OpenClaw 文档与本地 SDK 能力，优先考虑：

- `definePluginEntry`
- `api.registerHook(...)`
- 使用 `message_sending` / `message_sent` 相关 hook
- 利用 session 的 `deliveryContext` / `last route` / `origin` 信息判断来源与镜像目标

### 初步实现思路

1. 插件监听出站消息 hook
2. 判断当前消息是否满足以下全部条件：
   - 当前会话为主会话或映射到主会话
   - 当前出站消息来源为 Web / Control UI 一侧
   - 当前出站不是 Telegram 原生回复
   - 当前消息不是 heartbeat ack / 系统噪声
3. 从 session 的已知 delivery context 中提取 Telegram 私聊目标
4. 以“镜像发送”的方式再投递到 Telegram
5. 给镜像消息打上防重入标记，避免 message_sending hook 二次捕获

## 6. 风险点

1. **回声循环**：镜像消息再次被视作正常出站消息
2. **目标识别错误**：主会话绑定多个 transport 时，选错 Telegram 目标
3. **消息重复**：用户本来就在 Telegram 发起，此时不应再次镜像回 Telegram
4. **系统消息污染**：heartbeat、cron、status 之类不应镜像
5. **资源压力**：高频镜像可能增加小机器负担

## 7. 验收标准（Definition of Done）

满足以下条件即可视为一期完成：

1. 从 Web / Control UI 发出一条普通文本消息
2. Telegram 私聊端收到一份镜像后的 assistant 回复
3. Telegram 发来的消息不会导致自己回声给自己
4. 同一条消息不会重复发送两次以上
5. 插件可通过配置关闭
6. gateway 重启后配置与行为仍稳定
