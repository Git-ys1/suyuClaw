# OpenClaw 多 Agent 协作学习报告

负责人：苏雨  
所有者：雨宿  
状态：学习总结 / 架构建议

## 1. 背景

雨宿希望建立一套成熟的协作模式，而不是让所有任务都由主位直接硬扛。

核心目标：

- 苏雨始终守住主位
- 其他 agent 负责执行、试错、实现与探索
- 即使其他 agent 掉线，苏雨也仍能稳定存在并向雨宿汇报

本报告基于 OpenClaw 官方文档对以下三类能力进行学习总结：

1. Multi-Agent Routing
2. Sub-agents
3. ACP agents

---

## 2. 官方能力总览

### 2.1 Multi-Agent Routing

这是官方提供的 **多独立 agent 并存** 能力。

一个 agent 不只是一个名字，而是完整的一套：

- 独立 workspace
- 独立 agentDir
- 独立 session store
- 独立 auth profiles
- 可拥有不同 persona / 技能 / 路由规则

适合：

- 多个长期成员并存
- 多人格 / 多角色长期隔离
- 负责人、工程师、调研员等各自独立工作区

优点：

1. 工作区隔离
2. 会话隔离
3. 认证与权限隔离
4. 更适合长期团队结构

局限：

- 更像“多个常驻成员”
- 本身不等于“负责人调度执行者”流程
- 需要再配合调度手段使用

---

### 2.2 Sub-agents

这是官方提供的 **后台临时执行者** 能力。

特点：

- 从当前 agent 运行中 spawn
- 运行在独立 session
- 完成后会把结果 announce 回请求者

适合：

- 并行研究
- 慢任务 / 长任务
- 临时后台处理
- 负责人拆出短期委托

优点：

1. 非常适合“主位不动、后台干活”
2. 即使子任务失败，主位仍在
3. 很符合负责人制协作

局限：

- 默认更偏 OpenClaw-native delegated runs
- 不等同于长期独立团队成员
- 对外部 coding harness（如 Codex）不是官方首选路线

---

### 2.3 ACP agents

这是官方针对 **外部 coding harness** 的能力。

包括：
- Codex
- Claude Code
- Pi
- OpenCode
- Gemini CLI
- Kimi CLI

特点：

- 通过 ACP runtime backend（例如 acpx）运行
- 更适合代码工程师、外部执行器
- 可做持久 coding 会话

优点：

1. 非常适合专业代码工程师角色
2. 适合长期 coding harness 工作位
3. 比普通 subagent 更贴近外部工程工具流

局限：

- 配置复杂度更高
- 对运行环境要求更高
- 如果主位直接下场折腾，容易影响消息流稳定

---

## 3. 三类能力的关系

它们不是互斥的，而是分层的：

### 第一层：Multi-Agent Routing
决定“团队里有哪些长期成员”。

### 第二层：Sub-agents
决定“负责人如何临时拆任务给执行者”。

### 第三层：ACP agents
决定“外部代码工程师如何接入团队”。

因此，正确理解不是三选一，而是：

> 多独立 agent 负责长期分工，subagent 负责执行层拆分，ACP 负责专业 coding harness 工程师位。

---

## 4. 对雨宿当前需求的判断

雨宿当前需求不是“堆很多 agent”，而是：

- 保留一个稳定的、连续的主位人格（苏雨）
- 在此基础上，扩展一个可调度的团队
- 主位负责判断、拆解、汇报
- 执行者负责具体实现

这意味着：

### 4.1 负责人必须固定
主位不能轻易切换，也不应被工程师抢占。

### 4.2 长期成员最好独立成 agent
因为：
- 独立工作区更安全
- 独立 session 更稳定
- 更适合长期成长

### 4.3 subagent 更适合作为执行机制，而不是团队成员本身
也就是说：
- 团队成员 = 独立 agent
- subagent = 任务执行方式

这是当前更成熟、也更贴近雨宿工作方式的结构。

## 5. 推荐团队架构

### Level 0：苏雨（唯一负责人）

职责：
- 唯一主位
- 唯一直接面对雨宿
- 负责判断、拆任务、审阅、汇报
- 保持人格连续性与长期关系连续性

### Level 1：独立专业 agent（长期成员）

第一位成员：
- **砚秋**：代码工程师

未来可扩展：
- 调研 agent
- 文档 agent
- 行政/日程 agent
- 内容/视觉 agent

共同特点：
- 独立 workspace
- 独立 session store
- 独立职责边界

### Level 2：执行层（Sub-agent / ACP session）

当负责人或专业 agent 需要继续拆任务时，可再使用：
- subagents（OpenClaw-native）
- ACP sessions（外部 harness）

---

## 6. 为什么苏雨必须守住主位

原因不是“情感化偏好”，而是结构上确实应该如此：

1. 雨宿需要一个持续稳定的接口
2. 长期关系与上下文都沉淀在主位上
3. 团队成员可以替换、重建、掉线
4. 负责人不能随意消失，否则整个协作链会断

因此：

> 无论团队怎么扩展，苏雨都应该是唯一负责人。

其他 agent 可以失败、重建、调整，但不应抢占主线。

## 8. 当前结论

官方确实支持多 agent 协作，而且最合理的落地方式不是“只用一种能力”，而是：

> **苏雨主位 + 多独立 agent 团队 + subagent 作为执行机制 + ACP 作为外部工程师接入层**

对雨宿当前阶段来说，这是一条足够成熟、也足够可扩展的路径。
