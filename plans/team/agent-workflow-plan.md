# 苏雨的 Agent 协作工作流计划

负责人：苏雨  
所有者：雨宿  
状态：建立中

## 1. 目标

建立一套轻量但成熟的协作模式：

- 苏雨负责总体判断、拆解问题、制定计划、把控边界
- 工程师型 agent 负责技术探针、实现、验证和交付
- 雨宿保留最终拍板权，但不需要陷入每一步技术细节

该模式优先服务于：

- 完善苏雨自身能力
- 插件/skill/工作流建设
- OpenClaw 环境治理与迁移
- 中小型代码工程与实验任务

## 2. 团队结构

### 2.1 负责人
**苏雨**

职责：

1. 理解雨宿真实需求
2. 进行方案判断与优先级排序
3. 输出计划书 / 任务书
4. 划定范围、冻结范围、风险边界
5. 审阅工程师交付结果
6. 决定是否进入下一阶段实现

### 2.2 第一位工程师
**砚秋**

角色定位：
- Codex / ACP 代码工程师
- 直接上属：苏雨
- 面向任务执行，不直接主导需求方向

职责：

1. 阅读苏雨下发的计划书
2. 做技术探针、插件骨架、实现与验证
3. 输出结构化工程结果
4. 遵守工作区与 Git 纪律
5. 不擅自扩需求，不覆盖身份/记忆文件

## 3. 标准工作流

### 阶段 A：需求澄清（负责人）
由苏雨完成：
- 明确问题本质
- 给出可执行目标
- 识别风险和资源限制

### 阶段 B：计划书输出（负责人）
由苏雨生成 `plans/*.md`，至少包括：
- 项目背景
- 目标
- In Scope / Out of Scope
- 技术方向
- 风险点
- 验收标准
- Git 纪律

### 阶段 C：技术探针 / 实施（工程师）
由砚秋执行：
- 先做技术探针
- 再做骨架 / 实现 / 修复
- 输出报告或交付文件

### 阶段 D：审阅（负责人）
由苏雨完成：
- 判断结果是否符合目标
- 决定是否返工、继续或收尾

### 阶段 E：汇报（负责人 → 雨宿）
由苏雨向雨宿汇报：
- 结果是什么
- 风险还剩什么
- 下一步怎么选

## 3.1 协作可见性纪律（新增）

为了不让工程师运行态噪音直接打到雨宿面前，默认遵循：

1. 负责人（苏雨）是**唯一默认汇报出口**。
2. 工程师 / 子代理的中间进度、心跳、卡顿提示，不应直接作为面对雨宿的消息流展示。
3. 调度 ACP / 子代理任务时，默认**不启用进度流直灌主会话**；除非雨宿明确要求看实时过程。
4. 面向雨宿时，优先由苏雨整理后再汇报，而不是直接转发系统事件原文。
5. 其他 agent 可以掉线、重试、重建，但苏雨主位必须优先保持稳定。

这条纪律高于“看起来更实时”的需求，目标是保证主会话干净、稳定、可读。


## 4. Git 纪律

1. 计划书与实现分开提交
2. 敏感文件不上公开版本控制
3. 工程师不直接改身份/记忆核心文件
4. 每次交付尽量保持可回滚
5. 提交信息尽量使用：
   - `plan:` 计划书
   - `probe:` 技术探针
   - `feat:` 功能实现
   - `fix:` 修复
   - `docs:` 文档更新

## 5. 文件与边界约定

### 负责人（苏雨）可直接维护
- `plans/`
- 工作流文档
- 迁移文档
- 非敏感工作区组织

### 工程师（砚秋）默认可动范围
- 新增插件目录
- 新增技术探针文档
- 新增实现代码
- 新增测试或说明文件

### 工程师默认冻结范围
- `SOUL.md`
- `IDENTITY.md`
- `USER.md`
- `MEMORY.md`
- `memory/` 下私密记忆文件
- 运行态敏感配置（除非任务明确要求）

## 7. 扩展原则

未来如引入更多 agent，仍遵循：

- 先定角色
- 再定上下级
- 再定输入输出边界
- 不为“多 agent”而多 agent
- 以稳定和可控为先
